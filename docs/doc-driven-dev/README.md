# 📚 文档驱动开发 — 给开发者的快速上手指南

> **一句话总结**：在写任何代码之前，先把"产品到底在干什么"用大白话写成文档。AI 和人都靠这套文档来理解项目。

---

## 🚀 30 秒上手

### 新项目？跟 AI 脑暴

打开你的 AI 编程工具（Claude Code / Cursor / Gemini），告诉它：

```
请使用 doc-driven-dev 这个 skill，帮我搭建项目文档体系。
```

然后回答它问你的 7 个问题就行了，它会自动生成整套文档。

### 已有项目？补文档

```
请使用 doc-driven-dev 这个 skill，帮我为当前项目补充文档。
先读一下 src/ 目录下的代码，然后按 skill 里的流程生成文档。
```

## 🔑 核心原则

### 1. 先写业务文档，后写技术文档

```
❌ 错误：上来就写 API 文档、数据库 Schema
✅ 正确：先说清楚"用户想干什么"，再说"怎么实现"
```

### 2. 业务文档用说人话的方式写

```
❌ "系统触发异步邮件发送队列，消费者模式处理..."
✅ "用户点发送后，邮件在后台排队，一般 3 秒内发出去"
```

### 3. 模块关联文档是安全网

改任何涉及多个模块的功能之前，**必须先看 `module-coupling.md`**。
这份文档告诉你：改了这里，哪里会受影响。

### 4. 复盘不是形式主义

遇到问题写复盘 → 提炼成避坑清单 → 下次改代码前先看清单。
这是一个闭环，不是走流程。

## 🔄 日常使用流程

### 改代码前
```
1. 读 docs/README.md → 找到对应的文档
2. 读业务文档 → 理解要做什么
3. 读 module-coupling.md → 确认不会影响别的
4. 读避坑清单 → 确认没有已知的坑
5. 然后再改代码
```

### 出问题后
```
1. 先修复问题
2. 写复盘文档 → docs/postmortems/YYYY-MM-DD_描述.md
3. 更新避坑清单 → docs/postmortems/README.md
```

### 加新功能时
```
1. 先在业务文档中写清楚新功能的用户场景
2. 更新 module-coupling.md → 新模块和旧模块怎么联动
3. 确认方案 → 再开始编码
```

## ❓ FAQ

**Q: 小改动也要读文档吗？**
A: 如果只是改 CSS 或文案，不用。但只要涉及业务逻辑或数据，就要读。

**Q: 文档过时了怎么办？**
A: 每次改完代码后，顺手更新对应的文档。AI 改完代码也会被要求更新文档。

**Q: 我只想快速看一下整个项目，先读哪个？**
A: 读 `docs/business/README.md`，10 分钟看完整个产品在干嘛。

**Q: 用什么 AI 工具都可以吗？**
A: 是的。SKILL.md 里的格式和方法论是通用的，Claude Code、Cursor、Gemini 都能用。只需要在项目根目录有 `CLAUDE.md`（Claude）或等价配置文件即可。
