# Skill: 文档驱动开发（Doc-Driven Development）

> **核心理念**：先写文档，后写代码。文档不是代码的附属品，而是整个项目的设计源头。
>
> **适用场景**：新项目启动、已有项目补文档、帮助团队建立文档习惯。

---

## 🎯 这个 Skill 解决什么问题

1. **项目前期只有模糊想法**，不知道从哪开始梳理
2. **AI 改代码改出 bug**，因为不知道模块之间有联动
3. **换人接手时一脸懵**，没有上下文、没有设计思路
4. **复盘踩坑全靠记忆**，同样的错误反复犯

---

## 📋 完整流程：3 个阶段

```
阶段 1：业务脑暴（和产品负责人对话）
  ↓ 输出：业务文档骨架
阶段 2：业务文档落地（细化每个模块和页面）
  ↓ 输出：完整业务文档
阶段 3：技术文档拆解（基于业务文档，面向开发）
  ↓ 输出：技术实现文档 + CLAUDE.md 配置
```

---

## 阶段 1：业务脑暴

### 目标
用非技术语言，把「这个产品到底在干什么」说清楚。

### 操作方法

**你对 AI 说的 Prompt：**

```
你是一个项目文档架构师。我不是开发者，我是产品负责人。
我接下来会告诉你我的项目是做什么的，你帮我用大白话梳理清楚。

先依次问我这 7 个问题（如果我没主动说清楚）：

1. 这个产品是做什么的？（一段话，像跟朋友介绍一样）
2. 用户从头到尾会经历什么流程？（从进来到完成，每一步）
3. 有哪些主要功能模块？（比如"用户管理"、"订单系统"、"支付"）
4. 哪些页面是用户最常用的？（花时间最多的 2-3 个页面）
5. 有没有什么硬性规则或限制？（必须遵守的约束）
6. 模块之间有没有联动关系？（改了 A 会不会影响 B）
7. 用的什么技术栈？（框架、数据库、部署方式）

每个问题我回答完，你用大白话帮我总结一下，确认你理解对了。
全部问完后，帮我生成 docs/business/README.md（业务全貌文档）。
```

### 输出物 — `docs/business/README.md`

这是整个文档体系中**最重要的一份文档**。看完它就知道整个产品在干嘛。

**标准格式：**

```markdown
# [项目名] 业务流程与用户意图文档

> 本文档是所有功能设计和代码修改的业务参考。每次改动前，先对照本文档确认是否符合业务目标。

---

## 1. 业务全貌

### 1.1 做什么
（一句话说清楚这个产品为谁解决什么问题）

### 1.2 核心流程
（用 ASCII 流程图画出主链路，从用户进来到完成）
```
步骤1 → 步骤2 → 步骤3 → ... → 完成
```

### 1.3 核心约束
| 约束 | 说明 |
|---|---|
| 约束1 | 为什么有这个约束，违反了会怎样 |

---

## 2. 数据分层

### 2.1 基础数据（很少变的）
```
实体
├── 字段1（说明）
├── 字段2（说明）
└── 字段3（说明）
```

### 2.2 业务数据（经常操作的）
```
实体
├── 字段1（说明）
└── 字段2（说明）
```

---

## 3. [模块名] 模块

### 3.1 核心概念
（这个模块是干什么的，在整个流程中处于什么位置）

### 3.2 生命周期
```
创建 → 使用 → ... → 归档
```

### 3.3 用户操作意图

#### [场景名称]
**场景**：什么时候会这样操作
**操作**：
1. 第一步
2. 第二步
3. 第三步
**设计要点**：
- 为什么这样设计
- 有什么注意事项

---

## N. 模块间关系
```
实体A (1)
  ├── 实体B (1)
  ├── 实体C (0..N)
  └── 实体D (0..N)
```
```

## 阶段 2：业务文档落地

### 目标
把脑暴的结果细化成完整的文档体系，每个模块、每个页面、每个交互都有文档。

### 文档目录结构

```
docs/
├── README.md                          # 文档索引（导航页）
├── business/                          # 业务文档（写给人看的）
│   ├── README.md                      # 业务全貌（阶段 1 的输出）
│   ├── module-coupling.md             # 模块间关联关系（防止改坏）
│   ├── [模块名]-module.md             # 每个功能模块一份
│   ├── page-[页面名].md               # 每个重要页面一份
│   └── spec-[功能名].md               # 复杂交互的详细规格（按需）
├── technical/                         # 技术文档（写给 AI 和开发者看的）
│   ├── architecture.md                # 系统架构总览
│   └── [功能名]-spec.md               # 具体实现细节
├── postmortems/                       # 复盘与避坑
│   ├── README.md                      # 复盘机制说明 + 索引 + 避坑清单
│   └── YYYY-MM-DD_简短描述.md          # 每次复盘一份
├── changelogs/                        # 变更日志
│   └── CHANGELOG.md
└── plans/                             # 设计方案存档
```

### 按优先级写文档

**第一批（骨架，必须先写）：**

1. `docs/README.md` — 文档索引导航
2. `docs/business/README.md` — 业务全貌（阶段 1 已完成）
3. `docs/business/module-coupling.md` — 模块关联（防止改坏）
4. `docs/postmortems/README.md` — 复盘机制 + 避坑清单

**第二批（血肉，核心页面和模块）：**

5. 每个重要页面 → `docs/business/page-[name].md`
6. 每个功能模块 → `docs/business/[name]-module.md`

**第三批（细节，按需补充）：**

7. 复杂功能 → `docs/technical/[name]-spec.md`
8. 复杂交互 → `docs/business/spec-[name].md`

### 各文档的标准格式

#### 模块关联文档 — `docs/business/module-coupling.md`

**这是防止"改一个坏一片"的核心文档。**

```markdown
# 模块间关联关系

> 任何涉及跨模块的功能改动，必须参考本文档确认不会破坏关联逻辑。

---

## 1. 实体关系图

```
实体A (1)
  │
  ├── 实体B (0..1)        说明
  │     └── 子数据[]       说明
  │
  ├── 实体C (0..N)         说明
  │     └── 引用字段        引用了实体B的某个数据
  │
  └── 实体D (0..N)         说明
```

---

## 2. 关联点 1：[模块A] → [模块B] 的 [什么联动]

### 数据流
```
模块A的数据
  ├── 某字段 ──┐
  └── 某列表   ├── 合并/过滤/计算
               ▼
           模块B需要的数据
```

### 规则
1. 规则1：什么情况下怎么处理
2. 规则2：什么情况下怎么处理

### 影响（改了这里，那里会怎样）
- **做操作X时**：Y 模块会受到什么影响
- **做操作Y时**：Z 模块会受到什么影响

---

## N. 模块依赖关系图
```
         ┌──────────┐
         │  模块A   │ ← 顶层
         └────┬─────┘
              │
     ┌────────┴────────┐
     │                 │
     ▼                 ▼
 ┌────────┐      ┌────────┐
 │ 模块B  │─────→│ 模块C  │
 └────────┘      └────────┘
```

## 跨模块操作的数据一致性

| 操作 | 涉及模块 | 一致性保证 |
|---|---|---|
| 操作1 | A + B | 怎么保证不出问题 |
```

## 📖 如何使用这些文档

### 要改某个功能时
1. 先读业务文档 → 理解用户目标和场景
2. 再读模块关联文档 → 确认改动会不会影响其他模块

2. 检查模块关联文档 → 新功能和现有模块怎么联动

## 阶段 3：技术文档拆解

### 目标
基于业务文档，为 AI 和开发者提供实现细节。

### 什么时候写技术文档

- **不要一上来就写技术文档**。先有业务文档，再拆技术文档。
- 以下情况需要写技术文档：
  - 算法/策略涉及多种情况（生成规则、匹配逻辑等）
  - API 端点需要明确入参和返回
  - 数据库 Schema 需要记录
  - 跨组件的状态管理
  - 复杂的第三方集成

### 技术文档标准格式 — `docs/technical/[feature]-spec.md`

```markdown
# [功能名] — Technical Design Document

Date: YYYY-MM-DD
Status: Draft / Approved
Module: `涉及的文件路径`

---

## 1. Design Goal

（这个技术方案要解决什么问题）

### Core Principles
1. 原则1
2. 原则2

---

## 2. 核心逻辑

（用表格、代码块、流程图详细说明算法或策略）

---

## 3. Edge Cases

| 场景 | 输入 | 处理后 | 行为 |
|------|------|--------|------|
| 正常情况 | xxx | yyy | 正常处理 |
| 异常情况 | xxx | yyy | 给提示 |

---

## 4. 交互流程

```
用户操作 → 触发什么
    │
    ▼
系统处理：
    ├── 调用 API: /api/xxx
    │   └── 返回数据
    ▼
界面更新
```

---

## 5. API Endpoints

```
GET /api/xxx?param=value

Response: {
  field: "value"
}
```

---

## 6. Implementation Files

| File | Role |
|------|------|
| `src/lib/xxx.ts` | 核心逻辑 |
| `src/app/api/xxx/route.ts` | API 接口 |
| `src/components/xxx.tsx` | UI 组件 |
```

## ✅ 改进措施
1. 编码规范变更（具体新规则）
2. 流程/工具变更（更新了哪个 workflow）
3. 文档补充（更新了哪份文档）

## 配置项目的 AI 规则

### 必须做的：在项目根目录配置 CLAUDE.md（或等价文件）

在项目根目录的 `CLAUDE.md`（Claude Code 用）或 `.cursorrules`（Cursor 用）中添加以下内容：

```markdown
## ⚠️ 强制规则：改代码之前必须先读文档

**在修改任何功能代码之前，必须先阅读相关的业务文档和技术文档。**

文档索引在 `docs/README.md`，按以下流程操作：

1. 读 `docs/README.md` → 找到要改的功能对应哪个业务文档和技术文档
2. 读 `docs/postmortems/README.md` 的避坑清单 → 检查当前改动是否涉及已知的坑
3. 读对应的业务文档（`docs/business/`）→ 理解用户目标、操作场景、边界状态
4. 读 `docs/business/module-coupling.md` → 确认改动会不会影响其他模块
5. 读对应的技术文档（`docs/technical/`）→ 理解当前实现细节
6. 然后再动手改代码

**不确定该读哪个文档时，先读 `docs/business/README.md`（业务全貌）。**

## 🔁 强制规则：出问题后必须写复盘

遇到以下情况时，**必须**写复盘文档并更新避坑清单：

1. 部署失败
2. 线上 bug
3. 数据问题
4. 花了超过 30 分钟排查的问题
5. 差点引入的重大隐患

复盘流程：
1. 写复盘文档 → `docs/postmortems/YYYY-MM-DD_描述.md`
2. 更新避坑清单 → `docs/postmortems/README.md`
3. 如果涉及流程改进 → 更新对应的 workflow
4. 如果涉及编码规范 → 更新对应的技术文档
```

## 📝 文档写作铁律

### 业务文档（给人看的）
1. **用说人话的方式写** — "用户想干什么"而不是"系统执行什么"
2. **用场景驱动** — 每个功能点都是：场景 → 操作步骤 → 设计要点
3. **用内心独白** — 写用户在那个场景下心里在想什么（加引号）
4. **穷举边界状态** — 用表格列出所有可能的情况，不要说"等等"
5. **ASCII 图优于文字** — 流程、关系、数据流都用 ASCII 图画

### 技术文档（给 AI 和开发者看的）
1. **精确到文件** — 每个功能涉及哪些文件、哪个 API、什么数据结构
2. **代码示例用真实代码** — 不用伪代码，用项目实际的写法
3. **Edge Case 必须用表格** — 输入 → 处理后 → 行为
4. **数据流画图** — ASCII 流程图展示数据从哪来到哪去

### 模块关联文档（防止改坏的核心）
1. **每个关联点单独一节** — 不要笼统地说"它们有关系"
2. **画数据流** — 数据从 A 怎么流到 B
3. **列影响** — 改了这里，那里会怎样（正向列举）
4. **画依赖方向** — 谁依赖谁、谁不知道谁的存在

### 复盘文档（出问题后写的）
1. **带上报错信息** — 原始错误日志，不要只描述现象
2. **三层归因** — 直接原因 → 深层原因 → 流程原因，缺一不可
3. **改进措施必须可执行** — 不写"以后注意"，写具体规则/流程变更
4. **更新避坑清单** — 每次复盘后提炼成一条避坑条目

### 通用铁律
1. **不写空话** — 每句话都要有信息量
2. **不用模糊词** — 不说"等等"、"诸如此类"、"一些"，要穷举
3. **表格优于段落** — 能用表格就用表格
4. **一份文档只回答一个问题** — 业务文档不讲代码，技术文档不讲用户意图
5. **文档之间用链接互相引用** — 形成网络，不重复写相同内容
